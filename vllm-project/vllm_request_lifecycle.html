请求全链路详细流程

按照处理顺序，逐步追踪每个关键环节的类、方法和数据变换

HTTP 请求接收 (FastAPI Router)
vllm/entrypoints/openai/api_server.py
FastAPI 应用注册路由 POST /v1/chat/completions。
请求体被解析为 ChatCompletionRequest (Pydantic model)。
路由函数将请求转发给 ServingChat 的 handler。
Chat Completion 处理 (ServingChat)
vllm/entrypoints/openai/serving_chat.py
ServingChat.create_chat_completion() 核心逻辑：
1. 参数验证：检查 model name、temperature、top_p 等
2. Chat Template：用 Jinja2 模板将 messages[] 转为 prompt 字符串
3. Tokenization：调用 tokenizer 将 prompt 转为 token_ids
4. 多模态处理：如有图片/音频，加载并编码为 feature tensors
5. 构建 SamplingParams：包含 temperature, top_p, max_tokens, stop 等
6. 结构化输出：如有 response_format，配置 GuidedDecodingParams
7. Tool Calling：如有 tools，设置 tool_call_parser
最终调用 self.engine_client.generate()
AsyncLLM 引擎入口
vllm/v1/engine/async_llm.py
AsyncLLM.generate() 是 API Server 进程内的引擎客户端。
1. 调用 add_request() 生成唯一 request_id
2. 通过 InputPreprocessor 对输入做最终预处理
3. 构建 EngineCoreRequest 对象
4. 通过 ZMQ socket 将请求发送到 EngineCore 进程
5. 启动 output_handler 异步循环，监听来自 EngineCore 的输出
EngineCore 接收与调度
vllm/v1/engine/core.py → vllm/v1/core/scheduler.py
EngineCore 运行在独立进程中，以 busy loop 持续运行：
1. step() → 调用 Scheduler.schedule()
2. Scheduler 决策：哪些请求进入本轮 batch（新请求 prefill vs 运行中请求 decode）
3. 调用 KVCacheManager.get_computed_blocks() 检查 prefix cache 命中
4. 调用 KVCacheManager.allocate_slots() 为请求分配 KV cache blocks
5. 生成 SchedulerOutput：包含 scheduled_requests、block_table、num_computed_tokens 等
6. Chunked Prefill：长 prompt 可能被分成多个 chunk 分批处理
模型执行分发 (Executor)
vllm/v1/executor/multiproc_executor.py
MultiprocExecutor.execute_model(scheduler_output)
1. 将 SchedulerOutput 分发到所有 GPU Worker 进程
2. 在 Tensor Parallel 模式下，每个 worker 获得相同的调度信息
3. 等待所有 worker 完成并收集结果
GPU Worker 执行
vllm/v1/worker/gpu_worker.py → gpu_model_runner.py
GPUWorker.execute_model() → GPUModelRunner.execute_model()
1. 准备输入：将 token_ids, positions 组装为 GPU tensors
2. Attention Metadata：构建 FlashAttention/FlashInfer 的 metadata
3. CUDAGraph：如果 batch size 匹配已捕获的 graph，使用 CUDAGraph 加速
4. Forward Context：设置 set_forward_context() 传递 attention backend 信息
5. 调用 model.forward(input_ids, positions, ...)
模型前向传播 (Model Forward)
vllm/model_executor/models/llama.py (示例)
以 Llama 为例的典型前向传播路径：
1. embed_tokens(input_ids) → token embeddings
2. N × TransformerDecoderLayer：
   → RMSNorm → Attention(Q,K,V) + 残差连接
   → RMSNorm → MLP(gate_up_proj → SiLU → down_proj) + 残差连接
3. 最终 RMSNorm → hidden_states
Attention 层内部调用 PagedAttention kernel（FlashAttention / FlashInfer）
采样 (Sampling)
vllm/v1/worker/gpu_model_runner.py → vllm/v1/sample/
1. lm_head 线性层：hidden_states → logits (vocab_size 维度)
2. Logits 后处理：temperature scaling、top_p/top_k filtering、repetition penalty
3. 采样：根据 SamplingParams 进行 greedy/random/beam sampling
4. 如有 Speculative Decoding：验证 draft tokens，accept/reject
5. 输出 ModelRunnerOutput：包含 sampled_token_ids、logprobs
输出处理与返回
core.py → async_llm.py → serving_chat.py
1. EngineCore：update_from_output() 将新 token 追加到请求，检查 stop 条件
2. 构建 EngineCoreOutput，通过 ZMQ 发回 API Server 进程
3. AsyncLLM output_handler：接收输出，调用 Detokenizer 将 token_ids → text
4. 生成 RequestOutput，yield 给 ServingChat
5. ServingChat：构建 ChatCompletionResponse / ChatCompletionStreamResponse
   → 处理 reasoning_content、tool_calls、finish_reason
6. 非流式：等待所有 token 生成完毕，返回完整 JSON
7. 流式：每生成一个/几个 token，发送一个 SSE event

进程间通信 (IPC) 方式

API Server ↔ EngineCore
ZMQ Socket
API Server 和 EngineCore 通过 ZMQ IPC socket 通信。支持 many-to-many 拓扑：任何 API Server 可路由请求到任何 EngineCore。序列化使用 msgpack。
EngineCore ↔ GPU Workers
multiprocessing
EngineCore 通过 Python multiprocessing 管理 GPU Worker 进程。SchedulerOutput 通过共享内存传递，减少序列化开销。
Worker ↔ Worker (TP)
NCCL / CUDA
Tensor Parallel 模式下，Workers 之间通过 NCCL 进行 all-reduce 等集合通信操作，实现跨 GPU 的张量并行。
Client ↔ API Server
HTTP / SSE
标准 HTTP REST API。非流式返回完整 JSON；流式通过 Server-Sent Events (SSE) 逐 token 推送 delta。
进程数量公式： 总进程 = API Server (A, 默认=DP) + EngineCore (DP, 默认=1) + GPU Workers (N = DP × TP) + DP Coordinator (1 if DP>1, else 0)。 例如 4 GPU + TP=4：1 API Server + 1 EngineCore + 4 Workers = 6 个进程。

Prefill vs Decode 阶段对比

Prefill 阶段
输入：整个 prompt 的所有 token_ids
计算：一次性处理所有 prompt tokens
KV Cache：生成并存储所有 prompt 的 K/V
输出：第一个生成的 token
特点：compute-bound，高 GPU 利用率
优化：Chunked Prefill 可拆分长 prompt
Decode 阶段
输入：上一步生成的单个 token_id
计算：每步只处理 1 个新 token
KV Cache：追加 1 个 token 的 K/V
输出：下一个 token
特点：memory-bound，带宽瓶颈
优化：Continuous Batching 多请求共享 GPU
Continuous Batching： vLLM 的核心创新之一。不同于传统的 static batching（等所有请求都完成才释放资源），vLLM 在每个 decode step 动态地添加新请求和移除已完成请求，最大化 GPU 利用率。Scheduler 每一步都重新评估哪些请求应该被执行。
